# DATA_FETCH_CRITICAL_FIX.md — 数据获取致命问题修复

> 放到项目 `docs/`。本轮日志暴露了 3 个叠加问题,其中"LLM 编造数据"是红线级。
> 按优先级 P0 → P1 → P2 实施。每项都要在工作报告里说明根因和验证结果。

---

## P0-A(红线级)· 禁止数据缺失时 LLM 编造数据

### 问题
日志中,真实数据获取失败后,系统让 LLM 生成了一份"模拟分析报告",报告自述:
> "所有具体数值(价格、均线、MACD等)均为基于历史数据趋势的合理推演,并非当日真实交易数据"

**这是致命缺陷**:用户会拿一份编造数字的报告去做投资决策。比报错危险得多,也是合规灾难。

### 修复(必须根除)
1. **数据是分析的前置硬条件**:在进入任何 LLM 分析之前,先校验"是否成功取到真实K线数据"。取数失败 → **立即终止该股分析**,向用户返回明确错误,**绝不进入 LLM**。
2. 错误信息要具体且可操作,例如:
   > "❌ 无法获取 600519 的真实行情数据(数据源连接失败)。已尝试 Tushare、AKShare 均失败。请检查:① 网络连接 ② 数据源 API Key 配置 ③ 稍后重试。为保证分析基于真实数据,本次分析已终止。"
3. **审查所有 Agent 的提示词**:确保没有任何"数据不足时请基于历史趋势合理推演/模拟"之类的指令。把这类措辞**全部删除**,改为"若未提供真实数据,明确说明无法分析,不得编造任何数值"。
4. 在系统提示词层面加一条硬约束(所有分析 Agent 通用):
   > "你只能基于工具返回的真实数据进行分析。如果工具未返回有效数据,你必须明确告知'数据获取失败,无法分析',严禁编造、估算、推演或模拟任何价格、指标或成交量数值。"
5. 全局搜索代码里 "模拟"、"推演"、"假设"、"mock"、"fallback report" 等,清除"用假数据出报告"的代码路径。

### 验证
构造一个"数据源全部失败"的场景(如断网或填错 key),确认系统**报错终止**而非输出编造报告。把测试结果贴进工作报告。

## P0-B · 修通 Tushare 数据源(用户已付费,却未接通)

### 问题
日志:
```
❌ 备用数据源TUSHARE失败: 'DataSourceManager' object has no attribute '_get_tushare_stock_info'
```
`_get_tushare_stock_info` 方法**根本不存在**。用户配了 tushare token,但代码里 tushare 的取数分支是未实现的空壳,一降级就抛 AttributeError。这就是"配了 tushare 但数据体检仍失败"的根因。

### 修复
1. 实现缺失的 tushare 方法:`_get_tushare_stock_info`(股票基本信息)及对应的 K线获取方法。
2. Tushare K线用前复权:`pro.pro_bar(ts_code=..., adj='qfq', start_date=..., end_date=...)`(注意 ts_code 格式是 `600519.SH`,需做代码转换)。
3. 股票基本信息用 `pro.stock_basic()` 或 `pro.daily_basic()`。
4. **把 Tushare 设为 A股首选数据源,AKShare 作兜底**(Tushare 更稳定,且用户已付费)。当前默认是 AKShare,改为优先 Tushare。
5. 确认 token 从配置正确读取(日志显示"API Key来源: 环境变量",确认链路通)。
6. 字段映射:Tushare 返回列名(trade_date/open/close/high/low/vol)统一映射到项目标准英文列。

### 验证
用 tushare 单独取 600519 最近 K线,打印最近5日收盘价,确认与同花顺前复权一致(~1250)。贴输出。

## P1-A · 修复交易日逻辑(周末/节假日取不到数据)

### 问题
日志:`原始输入: 2026-06-21 至 2026-06-21`。6月21日是**周日**,股市不开盘,且首尾同一天 → 取不到任何数据。LLM 还把工具参数设成了首尾同日。

### 修复
1. **结束日自动回退到最近交易日**:若目标日是周末/节假日,回退到最近一个已收盘交易日(如周日→上周五)。可用交易日历(akshare `tool_trade_date_hist_sina` 或 tushare `trade_cal`)判断。
2. **取数区间要足够宽**:技术分析需要历史序列,start_date 应回溯足够天数(算 MA60 至少要 60+ 个交易日,建议默认回溯 ≥ 250 个自然日),**绝不能 start==end**。
3. 检查市场分析师工具调用:当前 LLM 传了 `start_date=end_date=2026-06-21`,要在工具内部强制纠正为合理区间(即使 LLM 传了错的,工具也要兜底扩展)。
4. 当日盘中数据不完整问题:明确用"最近已收盘交易日"为基准。

### 验证
在周末/非交易日运行,确认能正确取到最近交易日及之前的历史数据。

## P1-B · AKShare 网络错误重试 + 干净降级

### 问题
`('Connection aborted.', RemoteDisconnected('Remote end closed connection without response'))` — AKShare 免费源网络被掐断,偶发但常见。

### 修复
1. AKShare 取数加**指数退避重试**(retry 3 次,间隔 1s/2s/4s),覆盖 RemoteDisconnected/Timeout/ConnectionError。
2. 重试仍失败 → 干净降级到 Tushare(P0-B 修好后这条路才通)。
3. 降级链清晰:Tushare(首选)→ AKShare(重试)→ 明确报错终止(P0-A,不编造)。
4. 日志记录每次重试和降级,便于排查。

## P2 · 消除降级噪音(motor / baostock / curl_cffi)

### 问题
日志有 `No module named 'motor'`、`BaoStock数据源不可用: 库未安装` 等告警。这些是可选依赖缺失,不致命,但刷屏且让人误以为出错。

### 修复
1. `motor` 是 MongoDB 异步驱动,本项目用 file 缓存,不需要 → 把相关 import 改为可选,缺失时静默或 DEBUG 级,不打 WARNING。
2. baostock 若要作为第三兜底源,在 requirements 加上;若不用,把"不可用"降为 DEBUG。
3. 这些不影响功能,优先级最低,但清理后日志干净,便于发现真问题。
