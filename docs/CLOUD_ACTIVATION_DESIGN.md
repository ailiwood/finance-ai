# CLOUD_ACTIVATION_DESIGN.md — 线上购买 + 自动发卡 + 激活验证 架构设计

> 放到项目 `docs/`。技术路线:网页购买 → 支付成功 → 云端自动签发激活码 → 用户激活 → 客户端离线验证。
> 核心目标:私钥永不离开服务器;每个普通激活码只能用1次;预生成10个永久激活码(运维用)。

---

## 一、为什么换这条路(根因)

离线 keygen 反复失败的根因是三层问题叠加:GUI打包 + 私钥分发 + 跨机一致性。
云方案从根上消除:
- 私钥只存云端(环境变量/密钥管理),永不进客户端、永不分发。
- 用户浏览器打开网页即可购买激活,无需任何本地工具。
- 设备码从客户端读、发到云端签名,不存在跨机不一致。

已核实:Cloudflare Workers 的 Web Crypto API 现已原生支持 Ed25519 签名,可直接在云端签发,复用现有 Ed25519 体系。免费额度(10万请求/天,D1 5GB)对激活场景用不完,近乎0成本。

---

## 二、整体架构

```
用户浏览器 → 激活网站(前端) → 云函数(后端API)
  输入设备码+付款                1. 校验支付/凭证
                                2. 查/写订单数据库
                                3. Ed25519 私钥签名(私钥只在云端)
                                4. 返回激活码
用户拿到激活码 → 客户端输入 → 客户端用内置公钥离线验签 → 激活
```
客户端被极度简化:只保留"公钥验签 + 设备码匹配 + 过期检查",删除一切本地签发/keygen/私钥。

## 三、技术选型(两方案,个人MVP推荐方案B)

### 方案A:Cloudflare Workers + D1(全自动一站式)
- 优点:免费额度大、Web Crypto 原生 Ed25519、全球快。
- 缺点:**支付要自己接** —— Cloudflare 不管收款,你需自己申请支付宝/微信商户号。个人申请商户号通常需营业执照,门槛高。适合有营业执照、想做全自动的情况。

### 方案B(推荐个人MVP):发卡平台卖"凭证码" + 云函数换"激活码"
关键洞察:你真正需要的"自动收款+自动发货"有现成SaaS(自动发卡平台,它们自带聚合支付资质),不必自己造支付。但发卡平台发的是固定卡密,不能绑设备码 —— 所以拆成两段:
```
第1段(购买,用发卡平台):
  用户付款 → 平台自动发一个"购买凭证码"(从你预先上传的库里发,一次性)

第2段(激活,用你的云函数):
  用户打开激活网页 → 输入"凭证码" + 客户端设备码
  → 云函数校验凭证码未用过 → 用私钥签发绑定该设备码的激活码 → 标记凭证码已用
  → 返回激活码 → 用户复制到客户端激活
```
- 支付靠发卡平台(有资质、自动收款),设备绑定靠你的云函数(私钥安全)。
- 个人开发者门槛最低、最快上线。

> 有营业执照可走方案A把支付也并入云函数做全自动。MVP阶段方案B更现实。

## 四、云端数据库设计

```sql
-- 购买凭证码(预生成上传发卡平台)
CREATE TABLE vouchers (
  voucher_code TEXT PRIMARY KEY,
  status TEXT DEFAULT 'unused',   -- unused / used
  used_at TEXT,
  bound_device TEXT,
  issued_license TEXT
);
-- 激活记录
CREATE TABLE activations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  device_code TEXT,
  license_key TEXT,
  level TEXT,                      -- pro / permanent
  created_at TEXT,
  voucher_code TEXT
);
```

### 普通激活码"只能用1次" 如何保证
- 一次性体现在**凭证码**上:兑换时检查 status,unused 才签发并立即置为 used。
- 同一凭证码再兑 → 查到 used → 拒绝。
- 加上设备码绑定,激活码只在该设备有效。
- 并发防护:用数据库事务/条件更新(UPDATE ... WHERE status='unused')保证不被并发兑换两次。

## 五、10个永久激活码(运维用)

云函数提供受管理员密钥保护的签发接口:
```
POST /admin/issue-permanent
  Header: X-Admin-Secret: <只有你知道的管理员密钥,存云端环境变量>
  Body: { device_code: "...", note: "运维张三" }
  → 验证 Admin-Secret → 私钥签发 level="permanent", exp="9999-12-31"
  → 记入 activations(level=permanent)
```
两种永久码形态:
1. **绑设备永久码(推荐)**:运维各自提供设备码,逐个签发。最安全。
2. **万能永久码(应急)**:device 填特殊值 "MASTER",客户端验签时 level=permanent 且 device=="MASTER" 则跳过设备匹配。**泄露=任何设备可用,只给核心人、绝不外传。**

建议10个:8个绑设备给运维,2个万能码你自己应急保管。可写一个 scripts/issue_permanent.py 调 admin 接口批量签发。

## 六、客户端改动(大幅简化)

只保留验证:
1. 保留 src/core/license.py 的 verify_license(公钥验签+设备码匹配+过期)。
2. 删除/废弃:本地 keygen、GUI keygen、私钥文件及一切签发逻辑(私钥移云端)。
3. 激活页:显示设备码 + "点此购买"(跳发卡平台/激活网页)+ 激活码输入框 + 激活按钮。
4. 设备码沿用持久化UUID方案(device_id.py)。
5. 激活成功后状态持久化,下次启动复验放行。

## 七、防反编译 / 防源码泄露(关键点)

PyInstaller exe 可被反编译(pyinstxtractor解包+反编译pyc)。任何本地软件都无法100%防,目标是"提高门槛到不值得破解"。分层:
1. **核心验证上云(本方案已做)**:签名在云端,客户端只有公钥。即使反编译拿到公钥也签不出有效码 —— **把值钱的东西放在用户拿不到的地方**,这是最重要一层。
2. **字节码混淆**:用 pyarmor 加固核心模块(加密字节码、可绑机器)后再 PyInstaller 打包,大幅提高反编译门槛。
3. **关键分析逻辑也上云(进阶,可选)**:担心分析功能被白嫖,可把最核心编排做成云API,客户端只是壳。增加架构复杂度+服务器成本,MVP可不做。
4. **现实认知**:卖几十上百元的工具,做到第1+2层(云验证+pyarmor)已足够,破解成本高于直接买。不追求绝对安全。

## 八、安装包重新梳理

- 安装器:不碰激活,只装文件。
- 客户端:保留激活页 + verify_license(验签);删除本地签发/keygen/私钥。
- 重新打包:① 确认私钥绝不在包里 ② 核心模块 pyarmor 混淆 ③ 其余(Kronos全打包等)不变。

## 九、上线后的运维流程(你的日常)

```
日常售卖(全自动,你不用管):
  用户在发卡平台付款 → 自动收到凭证码 → 打开激活网页输凭证码+设备码 → 拿激活码 → 激活
你只需:① 定期往发卡平台补充凭证码库(用脚本批量生成上传) ② 收款到账

运维永久码(手动,一次性):
  运行 scripts/issue_permanent.py → 给运维人员签发永久码
```
