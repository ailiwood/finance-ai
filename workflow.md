# workflow.md — QuantSage 开发工作流

> 配合 CLAUDE.md 使用。本文件定义"怎么干活"，CLAUDE.md 定义"规则与定位"。

---

## 一、总体开发哲学

1. **复用 > 自研**：核心引擎用 TradingAgents-CN，不重造。我们的价值在"产品化外壳 + 插件 + 合规 + 打包"。
2. **能跑通 > 功能全**：V1 先交付"装得上、配得好、跑得出报告"，再加功能。
3. **降级优先**：每个功能都要能在"无 GPU / 无付费数据 / 无网络"下尽量优雅退化。
4. **合规内建**：免责声明和措辞校验是流程的一部分，不是事后补丁。

---

## 二、标准开发循环（每个任务都走这个流程）

```
1. 读 → 看 CLAUDE.md 第7节"当前进度" + 相关源码 + 本文件
2. 计划 → 列出改动点，确认不触红线，确认许可证合规
3. 写 → 小步实现，配类型注解
4. 测 → 写/跑单元测试，特别测降级分支
5. 校 → 跑 /check-compliance（涉及UI/报告时必跑）
6. 提 → 清晰 commit，更新 CLAUDE.md 第7节"当前进度"
```

## 三、里程碑任务清单（按依赖顺序，不设时间）

> 不规定耗时，只规定**先后顺序与完成标准**。逐个里程碑推进，达成完成标准即进入下一个。
> **整体两大阶段：先把方案 A 做通（M1→M6），再做方案 B（M7）。** 方案 A 面向发烧友/懂行用户/作者本人快速本地部署；方案 B 面向普通用户、走向商用。

### 阶段一 · 方案 A（Docker，首要目标）

#### M3 · Kronos 插件

#### M4 · FinBERT 情绪插件

#### M5 · 报告 + 合规
- [ ] 报告模板（基本面/技术面/情绪面/风控 四段 + 综合结论）

#### M6 · 方案 A 可分发（阶段一终点）

## 四、Git 工作流

- **分支**：`main`（稳定）/ `dev`（集成）/ `feature/xxx`（功能）
- **Commit 规范**（Conventional Commits）：
  - `feat: 新增 Kronos 预测插件`
  - `fix: 修复无 GPU 时崩溃`
  - `compliance: 报告页脚补充免责声明`
  - `docs: 更新安装指南`
  - `chore: 登记新依赖许可证`
- **每次 commit 后**：若改动影响架构/阶段，更新 `CLAUDE.md` 第7节。

## 五、测试要求

| 类型 | 要求 |
|------|------|
| 单元测试 | 每个新函数/模块都要有；放 `tests/` |
| 降级测试 | **强制**：无 GPU、无网络、无 API Key、无付费数据 四种降级路径 |
| 合规测试 | 报告/UI 文案必须通过 `compliance` 扫描（无违规措辞、有免责声明）|
| 集成测试 | 端到端跑一只股票分析，验证报告完整 |

运行：`pytest tests/ -v`

## 六、引入新依赖检查清单

每次 `pip install` 新包前，Claude Code 必须确认：
1. [ ] 许可证是否可商用？（MIT/Apache/BSD ✅；GPL/AGPL 先问用户）
2. [ ] 是否真的必要？能否用现有依赖实现？
3. [ ] 登记到 `THIRD_PARTY_LICENSES.md`
4. [ ] 是否引入大体积依赖（如另一份 PyTorch）影响打包？

## 七、调试常见问题

| 问题 | 排查方向 |
|------|----------|
| Kronos 报 CUDA 错误 | 检查 PyTorch 是否 Blackwell 兼容版本；FP8 支持；显存是否溢出 |
| 数据源拉取失败/被限频 | 检查本地缓存是否生效；调用间隔；换源（AkShare→Tushare）|
| LLM 调用超时 | 检查 API Key；本地 Ollama 是否启动；端口；超时重试逻辑 |
| Docker GPU 不可用 | NVIDIA Container Toolkit 是否装；compose 是否配 `--gpus all` |
| 报告缺免责声明 | 跑 `/check-compliance`；检查 `report` 注入器是否被调用 |
